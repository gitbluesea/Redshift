Amazon Redshift는 기본적으로 모든 데이터를 AP(애플리케이션) 서버의 메모리로 한 번에 가져오지 않습니다.
대규모 데이터 분석에 최적화된 Redshift의 데이터 처리 및 전달 방식은 다음과 같습니다.
------------------------

## 1. 연산 및 가공은 Redshift 내부에서 처리
Redshift는 MPP(대량 병렬 처리) 아키텍처를 사용합니다. 수억 건의 데이터에 대한 정렬, 그룹화(GROUP BY), 조인(JOIN) 등 모든 무거운 연산은 AP 서버가 아닌 Redshift의 컴퓨팅 노드(Compute Node) 메모리에서 전부 완료됩니다. [1, 2, 3] 

## 2. AP 서버에는 '최종 결과물'만 전달
모든 연산이 끝난 후, SELECT 쿼리의 최종 결과 데이터(Result Set)만 리더 노드(Leader Node)를 거쳐 AP 서버로 전송됩니다. 즉, 10억 건의 로우 데이터 중 조건에 맞아 필터링된 100건만 결과로 나온다면, AP 서버의 메모리에는 그 100건만 로드됩니다. [2, 4] 

## 3. AP 서버의 메모리가 고갈될 수 있는 예외 상황
하지만 AP 서버의 애플리케이션 코드를 어떻게 작성하느냐에 따라 AP 서버의 메모리가 부족해지는 현상(OOM)이 발생할 수 있습니다.

* Fetch Size 설정이 없을 때: MyBatis, JPA 등 Java 프레임워크나 Python 클라이언트에서 SELECT * FROM 대용량테이블을 실행하면, 드라이버가 결과물 수백만 건을 한 번에 AP 서버 메모리로 전부 올리려고 시도합니다.
* 해결법: 대용량 데이터를 가져와야 할 때는 클라이언트 사이드에서 Fetch Size(Cursor Fetch)를 설정하여 데이터를 몇 천 건씩 쪼개어(Streaming) 메모리에 올리며 처리해야 합니다.

## 4. 대용량 데이터 추출을 위한 권장 방식
만약 AP 서버에서 가공할 목적으로 대량의 데이터를 빼내야 한다면, SELECT 후 애플리케이션으로 받는 방식은 지양해야 합니다. 대신 Redshift의 UNLOAD 명령어를 사용하여 데이터를 AWS S3에 파일(Parquet, CSV 등) 형태로 먼저 내보낸 뒤, AP 서버가 S3에서 필요한 만큼만 나누어 읽도록 구현하는 것이 안전합니다.
AP 서버의 메모리 부하를 줄이기 위해 현재 어떤 언어나 프레임워크(예: Java/Spring, Python)를 사용 중이신지 알려주시면, 알맞은 Fetch 설정 코드 예시를 제공해 드리겠습니다.
