## 1. Amazon Redshift는 기본적으로 모든 데이터를 AP(애플리케이션) 서버의 메모리로 한 번에 가져오지 않습니다. 대규모 데이터 분석에 최적화된 Redshift의 데이터 처리 및 전달 방식은 다음과 같습니다.

### A. 연산 및 가공은 Redshift 내부에서 처리

 Redshift는 MPP(대량 병렬 처리) 아키텍처를 사용합니다. 수억 건의 데이터에 대한 정렬, 그룹화(GROUP BY), 조인(JOIN) 등 모든 무거운 연산은 AP 서버가 아닌 Redshift의 컴퓨팅 노드(Compute Node) 메모리에서 전부 완료됩니다. [1, 2, 3] 

### B. AP 서버에는 '최종 결과물'만 전달
모든 연산이 끝난 후, SELECT 쿼리의 최종 결과 데이터(Result Set)만 리더 노드(Leader Node)를 거쳐 AP 서버로 전송됩니다. 즉, 10억 건의 로우 데이터 중 조건에 맞아 필터링된 100건만 결과로 나온다면, AP 서버의 메모리에는 그 100건만 로드됩니다. [2, 4] 

### C. AP 서버의 메모리가 고갈될 수 있는 예외 상황
하지만 AP 서버의 애플리케이션 코드를 어떻게 작성하느냐에 따라 AP 서버의 메모리가 부족해지는 현상(OOM)이 발생할 수 있습니다.

* Fetch Size 설정이 없을 때: MyBatis, JPA 등 Java 프레임워크나 Python 클라이언트에서 SELECT * FROM 대용량테이블을 실행하면, 드라이버가 결과물 수백만 건을 한 번에 AP 서버 메모리로 전부 올리려고 시도합니다.
* 해결법: 대용량 데이터를 가져와야 할 때는 클라이언트 사이드에서 Fetch Size(Cursor Fetch)를 설정하여 데이터를 몇 천 건씩 쪼개어(Streaming) 메모리에 올리며 처리해야 합니다.

### D. 대용량 데이터 추출을 위한 권장 방식
만약 AP 서버에서 가공할 목적으로 대량의 데이터를 빼내야 한다면, SELECT 후 애플리케이션으로 받는 방식은 지양해야 합니다. 대신 Redshift의 UNLOAD 명령어를 사용하여 데이터를 AWS S3에 파일(Parquet, CSV 등) 형태로 먼저 내보낸 뒤, AP 서버가 S3에서 필요한 만큼만 나누어 읽도록 구현하는 것이 안전합니다.
AP 서버의 메모리 부하를 줄이기 위해 현재 어떤 언어나 프레임워크(예: Java/Spring, Python)를 사용 중이신지 알려주시면, 알맞은 Fetch 설정 코드 예시를 제공해 드리겠습니다.

## 2. Shell 스크립트(Bash 등)에서 Amazon Redshift 데이터를 조회할 때는 보통 psql 클라이언트나 aws redshift-data CLI를 사용합니다. 조회 결과가 수백만 건 이상으로 크다면, Shell 스크립트 실행 장비(AP 서버)의 메모리 고갈과 쿼리 타임아웃을 막기 위해 다음과 같은 방식으로 작성해야 합니다.

### A. UNLOAD 명령어로 S3에 내보내기 (가장 권장)
데이터 양이 매우 많다면 Shell 스크립트에서 직접 데이터를 받지 마세요. Redshift 내부에서 S3로 직접 데이터를 병렬 추출(UNLOAD)하게 만든 뒤, 스크립트에서는 S3 파일만 다운로드하는 것이 메모리와 속도 측면에서 가장 안전합니다.

```
#!/bin/bash
# Redshift 접속 정보 및 S3 경로 설정
export PGPASSWORD="your_password"
REDSHIFT_HOST="://amazonaws.com"
DB_NAME="your_db"
USER_NAME="your_user"
S3_BUCKET="s3://your-bucket-name/unload_data/"
IAM_ROLE="arn:aws:iam::123456789012:role/YourRedshiftRole"

# UNLOAD 쿼리 실행 (S3로 데이터 직접 전송)
psql -h "$REDSHIFT_HOST" -p 5439 -d "$DB_NAME" -U "$USER_NAME" -c "
UNLOAD ('SELECT * FROM large_table WHERE created_at >= \'2026-01-01\'')
TO '$S3_BUCKET'
IAM_ROLE '$IAM_ROLE'
FORMAT AS CSV
ALLOWOVERWRITE;
"
```

### 필요 시 S3에서 추출된 데이터를 로컬로 다운로드 (메모리를 쓰지 않고 디스크로 바로 저장)
```
aws s3 cp "$S3_BUCKET" ./local_download_dir/ --recursive
```

### B. psql 스트리밍 모드 활용 (텍스트 파일로 직접 저장)
S3를 거치지 않고 Shell 스크립트가 실행 중인 서버의 파일로 직접 저장해야 한다면, psql 실행 시 결과를 메모리에 쌓지 않고 곧바로 디스크 파일로 흘려보내도록(Streaming) 옵션을 주어야 합니다.

```
#!/bin/bash

export PGPASSWORD="your_password"
REDSHIFT_HOST="://amazonaws.com"
DB_NAME="your_db"
USER_NAME="your_user"

# -A (정렬 안 함), -q (메시지 숨김), -t (헤더 제외) 옵션 조합# 쿼리 결과를 화면에 출력하거나 변수에 담지 않고, 곧바로 파일(> output.csv)로 리다이렉트
psql -h "$REDSHIFT_HOST" -p 5439 -d "$DB_NAME" -U "$USER_NAME" \
     -A -q -t \
     -c "SELECT id, name, value FROM large_table;" > output.csv

echo "다운로드 완료: $(wc -l < output.csv) 라인"
```

* 주의: RESULT=$(psql ...) 처럼 쿼리 결과를 Shell 변수에 대입하면 안 됩니다. 변수에 할당하는 순간 데이터가 Shell 프로세스의 메모리에 통째로 올라가 서버가 멈출 수 있습니다. 반드시 > filename.csv 형태로 디스크에 바로 쓰세요.

### C. Cursor(커서)를 사용한 분할 조회
반드시 스크립트 내에서 데이터를 소량씩 읽어 가며 실시간 Loop 처리를 해야 한다면 SQL 커서를 선언하여 쪼개어 가져와야 합니다.

```
#!/bin/bash

export PGPASSWORD="your_password"
CONN_STR="-h ://amazonaws.com -p 5439 -d your_db -U your_user"

# 하나의 트랜잭션 안에서 커서를 열고 1000건씩 반복 FETCH
psql $CONN_STR -A -t -c "
BEGIN;
DECLARE my_cursor CURSOR FOR SELECT id, name FROM large_table;
FETCH FORWARD 1000 FROM my_cursor;
-- (실제 구현 시에는 루프를 돌며 FETCH를 반복하고 CLOSE; COMMIT; 처리 필요)
"
```

다만 Shell 스크립트 내부에서의 대용량 Cursor Loop 제어는 문법이 복잡하므로, 2번 방식으로 파일로 떨군 후 while read로 파일을 한 줄씩 읽는 것이 훨씬 직관적입니다.
